RevTurbine is web SaaS monetization in a single SDK: placement decisioning, entitlements, and conversion experiments — without a code deploy for every change. It's **additive** — if RevTurbine is disconnected, misconfigured, or every placement is paused, your product keeps working and slots simply render nothing.

This page gets you from zero to a rendered placement in local mode. For the detailed walkthrough see the [Quickstart](/getting-started/quickstart/); to pick a framework or server integration, jump to [Choose your integration path](#choose-your-integration-path).

## Why RevTurbine

<CardGrid>
  <Card title="Decisioning, not just rendering">
    RevTurbine optimizes for your revenue outcomes and a great end-user UI. It decides — caps, cooldowns, eligibility, priority across competing placements, and ML-scored propensity for conversion, expansion, and retention nudges. One `getPlacement()` call returns the winner. No client-side orchestration, no homegrown segment logic.
  </Card>

  <Card title="Entitlements and placements in one engine, linked to billing">
    Most web SaaS apps end up with three disconnected systems: entitlement access control, upgrade UX, and a billing provider. `getPlacement()` surfaces usage warnings, trial nudges, and other lifecycle prompts tied to billing state; `checkEntitlement()` confirms or denies feature access. Pair them to gate a feature *and* surface the customer-specific upgrade prompt that converts it. Like RevenueCat for SaaS.
  </Card>

  <Card title="PM- and marketer-controlled, dev-decoupled">
    Drop a slot once. Your PM, growth, marketing, or CRO team changes content, targeting, and CTAs in the studio — no engineering tickets, no deploys. UI path resolvers keep the navigation contract in your code; everything else iterates without you.
  </Card>
</CardGrid>

## What you'd otherwise build

| You want to… | Without RevTurbine | With RevTurbine |
|---|---|---|
| Show an upgrade banner to free users on day 7 | Cron + segment query + cap tracking + cooldown logic + content store | One placement in the studio |
| Block a feature with a contextual upgrade prompt | Custom gating + upgrade modal + plan lookup + Stripe redirect | `checkEntitlement()` + `getPlacement()` |
| Test three CTA variants for the same banner | Frontend deploy per change + custom analytics pipeline | Variants in the studio; RT picks the winner |
| Show a trial-ending nudge three days out | Email infra + in-app toast logic + dedupe | One trial placement; RT routes it to the surface |
| Gate a feature by plan tier, not just plan ID | Hard-coded plan checks, refactored every plan rename | `checkEntitlement(handle, { requiredTier })` |

## Quickstart — local mode, no backend

The SDK runs entirely client-side in `local_only` mode: ship an `exported_config.json` with your app and every decision happens offline — no API keys, no network calls.

<Steps>

1. **Install the SDK**

    `@revturbine/sdk` is published to the public npm registry — no auth token or registry configuration needed.

    <Tabs>
      <TabItem label="npm">
      ```bash
      npm install @revturbine/sdk
      ```
      </TabItem>
      <TabItem label="pnpm">
      ```bash
      pnpm add @revturbine/sdk
      ```
      </TabItem>
      <TabItem label="yarn">
      ```bash
      yarn add @revturbine/sdk
      ```
      </TabItem>
    </Tabs>

2. **Add an exported config**

    Export `exported_config.json` from the RevTurbine studio and place it in your `src/` directory. It contains your plans, entitlements, placements, targeting rules, surface templates, and **UI paths** — everything the SDK needs to run locally.

3. **Wrap your app in `RevTurbineProvider`**

    ```tsx title="src/App.tsx"
    import { RevTurbineProvider } from '@revturbine/sdk';
    import exportedConfig from './exported_config.json';
    import { useMemo } from 'react';

    function App() {
      const options = useMemo(() => ({
        localRuntime: { exportedConfig },
        user: {
          id: 'user_123',
          context: { plan_handle: 'starter' },
        },
        uiPathResolvers: {
          navigate_to_plans: () => { window.location.href = '/pricing'; },
          open_checkout_modal: (ctx) => { console.log('Checkout for plan:', ctx.plan_handle); },
          book_demo: (ctx) => { window.open(ctx.url ?? '/sales/demo', '_blank'); },
          custom_url: (ctx) => { if (ctx.url) window.location.href = ctx.url; },
        },
      }), []);

      return (
        <RevTurbineProvider options={options}>
          <YourApp />
        </RevTurbineProvider>
      );
    }
    ```

    <Aside type="tip">
    **UI path resolvers** connect placement CTAs to your app's navigation. When a user clicks "Upgrade Now" inside a placement, the SDK invokes *your* resolver for that `action_type` — so you keep full control of checkout flows, routing, and modals. See [What are UI paths?](/getting-started/quickstart/#what-are-ui-paths) for the full list.
    </Aside>

4. **Drop in a placement slot**

    ```tsx title="src/components/Dashboard.tsx"
    import { FixedSurfaceSlot } from '@revturbine/sdk';

    function Dashboard() {
      return (
        <div>
          <h1>Dashboard</h1>

          {/* RevTurbine renders an upgrade banner here — or nothing */}
          <FixedSurfaceSlot
            id="dashboard_top_banner"
            surfaceTemplateIds={["banner_placement"]}
          />

          {/* Rest of your dashboard */}
        </div>
      );
    }
    ```

5. **Run it**

    Start your dev server. The SDK evaluates targeting rules from the exported config and renders the matching placement for each slot. If no placement matches the current user context, the slot renders nothing — the additive principle at work.

</Steps>

## What the SDK can do

| Capability | Method | Description |
|---|---|---|
| **Placements** | `getPlacement()` | Render upgrade banners, modals, toasts, and buttons based on targeting rules |
| **Entitlements** | `checkEntitlement()` | Gate features, enforce usage limits, check credits and seats |
| **Usage tracking** | `updateUsage()` | Track consumption for quota meters and passive placement triggers |
| **Events** | `trackEvent()` | Behavioral signals for propensity scoring |
| **Interactions** | `dismiss()` / `snooze()` / `convert()` | CTA lifecycle tracking |
| **Trial status** | `getTrialStatus()` | Current trial state and countdown |
