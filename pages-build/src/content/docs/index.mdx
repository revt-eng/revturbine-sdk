## Why RevTurbine

<CardGrid>
  <Card title="Decisioning, not just rendering">
    RevTurbine optimizes for your revenue outcomes and a great end-user UI. RevTurbine decides — caps, cooldowns, eligibility, priority across competing placements, and ML-scored propensity for conversion, expansion, and retention nudges. One `getPlacement()` call returns the winner. No client-side orchestration, no homegrown segment logic.
  </Card>

  <Card title="Entitlements and placements in one engine, linked to billing">
    Most web SaaS apps end up with three disconnected systems: entitlement access control, upgrade UX, and a billing provider. `getPlacement()` surfaces usage warnings/limits, trial nudges and other lifecycle retention nudges linked to billing state. For gated features, RevTurbine pairs `checkEntitlement()` to confirm/deny entitlement access, e.g. gate the feature with `getPlacement()` to surface the customer and feature-specific gate to convert the upgrade. Like RevenueCat for SaaS.
  </Card>

  <Card title="PM or marketer-controlled, dev-decoupled">
    Drop a slot once. Your PM, growth team, marketing team or CRO consultant changes content, targeting, and CTAs in the studio — no engineering tickets, no deploys. UI Path resolvers keep the navigation contract in your code; everything else iterates without you.
  </Card>
</CardGrid>

## What you'd otherwise build

| You want to… | Without RevTurbine | With RevTurbine |
|---|---|---|
| Show an upgrade banner to free users on day 7 | Cron + segment query + cap tracking + cooldown logic + content store | One placement in the studio |
| Block a feature with a contextual upgrade prompt | Custom gating + upgrade modal + plan lookup + Stripe redirect | `checkEntitlement()` + `getPlacement()` |
| Test three CTA variants for the same banner | Frontend deploy per change + custom analytics pipeline | Variants in the studio, RT picks the winner |
| Show a trial-ending nudge three days out | Email infrastructure + in-app toast logic + dedupe | One trial placement, RT routes to the surface |
| Gate a feature by plan tier, not just plan ID | Hard-coded plan checks, refactored every plan rename | `checkEntitlement(handle, { requiredTier })` |

## Design Principle: Additive Only

RevTurbine is **additive** — it enhances your app but is never required for baseline UX. If RevTurbine is disconnected, misconfigured, or if all placements are paused, your product continues to work normally. Surface slots return "nothing to show" by default; your app renders its standard UI.

## Try It — Local Mode in 4 Steps

The SDK runs entirely client-side in `local_only` mode. No backend, no API keys, no network calls. Ship an `exported_config.json` with your app and everything works offline.

<Steps>

1. **Configure registry & install**

    The SDK is published to GitHub Packages. Add an `.npmrc` to your project root:

    ```ini title=".npmrc"
    @revt-eng:registry=https://npm.pkg.github.com
    //npm.pkg.github.com/:_authToken=${GITHUB_TOKEN}
    ```

    Then install:

    <Tabs>
      <TabItem label="npm">
      ```bash
      pnpm add @revt-eng/sdk
      ```
      </TabItem>
      <TabItem label="pnpm">
      ```bash
      pnpm add @revt-eng/sdk
      ```
      </TabItem>
      <TabItem label="yarn">
      ```bash
      yarn add @revt-eng/sdk
      ```
      </TabItem>
    </Tabs>

    <Aside type="caution">
    `GITHUB_TOKEN` must be a [personal access token](https://github.com/settings/tokens) with `read:packages` scope. Never commit it to source control.
    </Aside>

2. **Wrap your app with local runtime**

    ```tsx title="src/App.tsx"
    import { RevTurbineProvider } from '@revt-eng/sdk';
    import exportedConfig from './exported_config.json';
    import { useMemo } from 'react';

    function App() {
      const options = useMemo(() => ({
        localRuntime: { exportedConfig },
        user: {
          id: 'user_123',
          context: { plan_handle: 'starter' },
        },
        uiPathResolvers: {
          navigate_to_plans: () => { window.location.href = '/pricing'; },
          open_checkout_modal: (ctx) => { console.log('Checkout:', ctx.plan_handle); },
          book_demo: (ctx) => { window.open(ctx.url ?? '/sales/demo', '_blank'); },
          custom_url: (ctx) => { if (ctx.url) window.location.href = ctx.url; },
        },
      }), []);

      return (
        <RevTurbineProvider options={options}>
          <YourApp />
        </RevTurbineProvider>
      );
    }
    ```

    <Aside type="tip">
    The `exported_config.json` contains your plans, entitlements, placements, targeting rules, and **UI paths**. Export it from the RevTurbine dashboard.
    </Aside>

3. **Wire up UI path resolvers**

    **UI paths** connect placement CTAs to your app's navigation. When a user clicks "Upgrade Now" inside a placement, the SDK calls your resolver for that `action_type` — giving you full control over checkout flows, routing, and modals. See [What Are UI Paths?](/getting-started/quickstart/#what-are-ui-paths) for the full list of action types.

4. **Drop in a placement slot**

    ```tsx title="src/components/Dashboard.tsx"
    import { FixedSurfaceSlot } from '@revt-eng/sdk';

    function Dashboard() {
      return (
        <div>
          <h1>Dashboard</h1>
          <FixedSurfaceSlot
            id="dashboard_top_banner"
            surfaceTemplateIds={["banner_placement"]}
          />
        </div>
      );
    }
    ```

    The SDK evaluates targeting rules locally and renders the matching placement — or nothing if no placement applies. **Your app always works**, with or without RevTurbine.

</Steps>

## What Can the SDK Do?

| Capability | Method | Description |
|---|---|---|
| **Placements** | `getPlacement()` | Render upgrade banners, modals, toasts, and buttons based on targeting rules |
| **Entitlements** | `checkEntitlement()` | Gate features, enforce usage limits, check credits and seats |
| **Usage tracking** | `updateUsage()` | Track consumption for quota meters and passive placement triggers |
| **Events** | `trackEvent()` | Behavioral signals for propensity scoring |
| **Interactions** | `dismiss()` / `snooze()` / `convert()` | CTA lifecycle tracking |
| **Trial status** | `getTrialStatus()` | Current trial state and countdown |
